# 活力纪元 (Vitalife) 后端架构与接口文档

## 一、架构核心思想：黑板架构 (Blackboard Architecture)

面容、生理信号、报告单三大模块为**无状态的感知器 (Sensors)**，仅负责本地深度学习模型推理并将结构化结果落库；**Agent（智能体中枢）** 为核心大脑，仅在用户发起综合查询时唤醒，读取数据库多模态数据进行决策层后融合。

该架构解耦 “耗时的深度学习推理” 与 “复杂的 LLM 编排”，容错率极高（单一感知模块异常不影响 Agent 工作）。

## 三、核心数据流转说明

### 阶段 1：感知与入库 (无 Agent 参与)

**场景**：用户上传面部照片分析

1. Router 层：`api/routers/face.py` 接收 `POST /api/v1/face/analyze` 请求，提取 `user_id` 和图片；
2. Service 层：路由调用 `api/services/face_service.py`；
3. 模型推理：`face_service.py` 调用 `models_local/face_inference.py` 执行本地深度学习模型推理；
4. 数据库落盘：推理结果通过 SQLAlchemy 插入 `facecv_records` 表；
5. 返回前端：直接返回 200 与计算结果，流程结束。

### 阶段 2：融合与决策 (唤醒 Agent)

**场景**：用户点击 “生成综合报告” 或发起对话查询

1. Router 层：`api/routers/chat.py` 接收 `POST /api/v1/chat/query` 请求；
2. Service 组装上下文：`chat_service.py` 按 `user_id` 查询 MySQL 中面部、生理信号、报告单最新记录；
3. 唤醒 Agent：将查询结果拼装为 `context_data`，传入 `agents/MAS_GRAPH.py` 启动多智能体协作；
4. 决策生成：LangGraph 完成多模态数据融合与医疗推理，生成综合报告；
5. 落盘与返回：报告存入 `chat_history` 表，同时返回前端。

## 四、数据库设计 (MySQL 8.0+)

核心设计原则：**感知层写流水，认知层查最新**，通过 JSON 类型保证扩展性，联合索引保证查询效率。

### 0. 创建数据库

sql

```sql
CREATE DATABASE IF NOT EXISTS `vitalife_db` 
DEFAULT CHARACTER SET utf8mb4 
COLLATE utf8mb4_unicode_ci;

USE `vitalife_db`;
```

### 1. 用户基础信息表 (`users`)

```sql
CREATE TABLE `users` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `user_id` VARCHAR(50) NOT NULL UNIQUE COMMENT '业务全局唯一标识(如 U1001)，替代自增ID对外暴露',
    `username` VARCHAR(50) NOT NULL UNIQUE COMMENT '登录账号(手机号)',
    `password_hash` VARCHAR(255) NOT NULL COMMENT '加密后的密码',
    `name` VARCHAR(50) COMMENT '真实姓名',
    `gender` ENUM('male', 'female', 'other') COMMENT '性别',
    `actual_age` INT COMMENT '实际年龄(计算衰老指数的基准)',
    `medical_history` TEXT COMMENT '既往病史(自然语言文本)',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '账号注册时间',
    `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '最后修改时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户基础信息表';
```

### 2. 面部视觉感知记录表 (`facecv_records`)

sql

```sql
CREATE TABLE `facecv_records` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `user_id` VARCHAR(50) NOT NULL COMMENT '关联用户的 user_id',
    `image_path` VARCHAR(255) NOT NULL COMMENT '面部图片在OSS或本地的存储路径',
    `predicted_age` DECIMAL(5,2) COMMENT 'AI预估的生物学年龄(如 37.45)',
    `aging_index` DECIMAL(5,2) COMMENT '衰老指数(预估年龄 - 实际年龄)',
    `skin_analysis` JSON COMMENT '皮肤多维度评分字典(泛红、出油、毛孔等)',
    `fatigue_analysis` JSON COMMENT '疲劳扫描评分字典(左右眼、面部松弛等)',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '该次照片分析的时间',
    
    CONSTRAINT `fk_face_user` FOREIGN KEY (`user_id`) REFERENCES `users`(`user_id`) ON DELETE CASCADE,
    INDEX `idx_user_time` (`user_id`, `created_at` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='面部视觉单模态感知流水表';
```

### 3. 生理信号分析记录表 (`signal_records`)

sql

```sql
CREATE TABLE `signal_records` (
```

### 4. 报告单结构化解析记录表 (`report_records`)

sql

```sql
CREATE TABLE `report_records` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `user_id` VARCHAR(50) NOT NULL COMMENT '关联用户的 user_id',
    `image_path` VARCHAR(255) NOT NULL COMMENT '化验单原图存储路径',
    `summary` TEXT COMMENT '大模型对本页报告给出的初步一句话摘要',
    `lab_results` JSON COMMENT '提取出的EAV(实体-属性-值)结构化指标数组',
    `confidence` DECIMAL(3,2) COMMENT '本次OCR与信息抽取的总体置信度(0.00 - 1.00)',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '单据解析的时间',
    
    CONSTRAINT `fk_report_user` FOREIGN KEY (`user_id`) REFERENCES `users`(`user_id`) ON DELETE CASCADE,

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='体检报告OCR解析流水表';
```

### 5. 智能体综合对话记录表 (`chat_history`)



```sql
CREATE TABLE `chat_history` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `user_id` VARCHAR(50) NOT NULL COMMENT '关联用户的 user_id',
    `query_text` TEXT COMMENT '用户的自然语言提问(若是点击"一键生成报告"则为空或默认提示)',
    `agent_reply` TEXT NOT NULL COMMENT 'MAS智能体中枢生成的综合答复/医学报告',
    `health_score` INT COMMENT '多模态融合计算出的综合健康打分(0-100)',
    `context_used` JSON COMMENT '数据溯源字典：记录本次回答引用了哪些面部、信号、报告的记录ID',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '对话/报告生成时间',
    
    CONSTRAINT `fk_chat_user` FOREIGN KEY (`user_id`) REFERENCES `users`(`user_id`) ON DELETE CASCADE,
    INDEX `idx_user_time` (`user_id`, `created_at` DESC)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='Agent综合推理与用户对话历史表';
```

### 数据库设计亮点

1. **统一查询模式**：所有记录表均含 `user_id + created_at` 联合索引，Agent 查询最新数据时 `SELECT * FROM table WHERE user_id=? ORDER BY created_at DESC LIMIT 1` 可毫秒级命中；
2. **读写分离与高容错**：感知表独立写入，单一模块异常不影响 Agent 决策；
3. **幻觉溯源**：`chat_history.context_used` 记录数据来源 ID，便于定位 AI 结论异常原因。

## 五、API 接口文档 v3.1

### 全局约定

- Base URL: `http://<server-ip>:8000/api/v1`
- 通信协议: HTTP/HTTPS
- 数据格式：文本请求 `application/json`，文件请求 `multipart/form-data`
- 鉴权方式：暂通过 `user_id` 识别用户身份
- 通用响应结构:

```json
{
  "code": 200,             // 业务状态码
  "msg": "success",        // 状态描述信息
  "data": {}               // 具体业务数据
}
```

### 全局业务状态码规范

| 状态码 (Code) |             含义大类             |                  使用场景说明与前端处理建议                  |
| :-----------: | :------------------------------: | :----------------------------------------------------------: |
|      200      |          成功 (Success)          |           业务处理正常，前端直接提取 `data` 渲染。           |
|      400      |      参数错误 (Bad Request)      | 前端传参遗漏（如缺失 `user_id`），静默上报或 Toast 提示重试。 |
|      401      |      未授权 (Unauthorized)       |         `user_id` 不存在或密码错误，前端弹回登录页。         |
|      415      | 媒体类型错误 (Unsupported Media) |   上传不支持的文件格式（如 pdf），弹窗提示更换为 JPG/PNG。   |
|      422      |   业务逻辑阻断 (Unprocessable)   | 格式合法但业务不合规（如无人脸、PPG 伪影），弹窗引导用户重新操作（如 “光线太暗，请重拍”）。 |
|      500      | 服务器内部错误 (Internal Error)  | 显存溢出、数据库断连等，前端提示 “服务暂不可用，请稍后重试”。 |
|      504      |  AI 推理超时 (Gateway Timeout)   | 模型推理耗时过长，前端展示加载态（如 “AI 医生正在深度分析中，请稍候”）。 |
